iro treasurer website

I became the treasurer for the IRO in November of 2022, and my campaign promise was to make the club's finances transparent, so I made a website displaying Bank of America information. You can visit it here. Although I wish I could publish the repo on GitHub as I am very proud of it, it contains confidential banking information. If you want me to publish the webscraper, though, contact me and I will.

Unfortunately, Bank of America is secure [citation needed] and doesn't have an API. So, to get banking information automatically I had to use Selenium to webscrape. This required me to both sign-in to Bank of America automatically and automate the two-factor authentication. So, the webscraper first logs in to Bank of America, checks to see if two-factor authentication is required (it is), uses Google's OAuth 2.0 certification to log into the email account associated with the bank account, retrieves the authorization code, and then finally logs in. Once logged in, the scraper then scrapes the account's balance and recent transactions and displays them on the website in a readable manner.

When typing it out, designing this website sounded like it could take me a day, but this took me over two weeks to make because of unexpected hurdles. Bank of America really doesn't like it when you request over 20 authorization codes within a short succession and gets very angry at you and tries to lock you out of your own account. The Google OAuth 2.0 system can be quite annoying to work with when tokens and credentials conflict. Since I wrote the scraper in Python, I had to use Django to host the website to run the scripts in the background and update the website once they finished. Hosting the site on Heroku posed more challenges, as I had to figure out how to configure buildpacks that allowed Selenium to run and deal with a variety of issues including unsupported packages, annoying procfiles, and long deployment times.

twitter to lcd

As I mentioned in my about page, I have a passion for creating and automating random things on my Rasberry Pi for no reason. This project uses the Twitter API for python, tweepy, to live stream the most recent mentions for my account (or any account, for that matter). If a new mention is found, it displays who said it and the contents of the tweet; if not, it keeps displaying "waiting for mentions."

The hardware I used in this project was pretty simple - A Raspberry Pi (and a keyboard, mouse, monitor, and power source). Further, I connected an Arduino connected to the Pi with a USB cord, and connected a 16x2 LCD display to the Arduino. Since the Pi has a Wifi chip, I let the Pi communicate with the Twitter API, and then it sends the information using the USB serial protocol to the Arduino to then parse and display on the LCD.

The hardest part about this project was (surprisingly) not using Twitter's API, but making calls to Twitter's API in a "live stream" manner. Because I had a free developer account, it was difficult to "live stream" without using too many requests within a period of time (Twitter timed my account out many, many times ¯\_(ツ)_/¯ ), so the program checks for new mentions every five seconds or so. Further, most tweets are more than 32 characters, so I had to implement a scrolling method to display all of the text, which was somewhat of a challenge.

spotify control

With the running theme of using a Raspberry Pi, I also unnecessarily used a HW-504 joystick to control my Spotify music. The Pi is constantly on, and whenever the joystick button is pressed down, music either stops/starts playing. Tilting the joystick left skips forward, right skips backwards, up inreases the volume, and down decreases the volume.

This project was made much easier with very straightforward documentation from the Spotify API/Spotipy's readthedocs, and using the joystick was quite easy, as well. The only minor hiccup was finding ways to avoid the "noise" of the joystick when it's not been moved. Even when it's stationary and no one has touched it, the joystick still sometimes "moves". So, finding a way to avoid the unnecessary noise and determine what constitutes an actually human inputted input was the hardest part.
